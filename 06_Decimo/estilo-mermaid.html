8  Estilo diagramas Mermaid

En Quarto los diagramas se escriben con ```{mermaid} (no ```mermaid). Los bloques funcionales renderizan automáticamente; los bloques con eval: false muestran el código sin renderizar (útil para ejemplos de sintaxis).

8.0.1 Cuándo usar un diagrama

  • El concepto tiene partes que se relacionan entre sí
  • Hay un flujo o secuencia de pasos con decisiones
  • Hay una jerarquía o clasificación
  • El texto solo no logra que el estudiante visualice la idea
  • No usar diagramas para decorar — solo cuando aporten comprensión real

8.0.3 Dirección del flowchart

Regla: usar TD para jerarquías y clasificaciones, LR para procesos y secuencias encadenadas.

8.0.10 Mindmap — mapa mental

Usar para mostrar la estructura de un tema con sus subtemas. No soporta classDef — usar cuando la estructura importa más que el color.

8.0.11 Timeline — línea de tiempo

Usar para mostrar evolución histórica, fases de un proceso o etapas de desarrollo. No soporta classDef.

8.0.12 Sequence diagram — interacción entre elementos

Usar para mostrar cómo interactúan partes de un sistema o los pasos de un proceso con actores.

8.0.13 Journey — recorrido o experiencia del estudiante

La escala es de 1 a 5 — representa qué tan fácil o satisfactoria es cada etapa para el estudiante. Usar para mostrar el recorrido de aprendizaje de un tema o proceso.

8.0.14 Pie chart — proporciones y porcentajes

Usar para mostrar distribución de partes de un todo. El title es obligatorio para que el estudiante entienda qué representa el gráfico.

8.0.15 Quadrant chart — clasificación en cuadrantes

Usar para comparar elementos según dos dimensiones simultáneamente. Los ejes deben estar claramente etiquetados con términos que el estudiante entienda. Los valores van de 0 a 1 en ambos ejes.

8.0.16 XY Chart — datos numéricos con barras y líneas

Usar para mostrar evolución de datos numéricos a lo largo del tiempo o fases. La line encima de la bar ayuda a visualizar la tendencia. El title y las etiquetas de los ejes son obligatorios.

8.0.17 GitGraph — flujos con ramificaciones o caminos alternativos

Usar para mostrar ramificaciones de un proceso, versiones de un concepto o caminos alternativos que luego convergen.

Tipos de commit disponibles:

  • NORMAL — paso estándar (default)
  • HIGHLIGHT — paso importante o crítico
  • REVERSE — paso que revierte algo

8.0.18 Texto en nodos

  • Máximo 5 palabras por nodo
  • Sin abreviaciones
  • Si el texto es largo usar comillas: A["texto más largo aquí"]
  • Para caracteres especiales usar comillas: A["¿Qué es la energía?"]
  • Para negrita en nodo usar markdown strings: A["\texto`“]`

8.0.19 Gotchas

  • No usar end en minúscula dentro de un nodo — rompe el diagrama; usar End o END
  • No iniciar un nodo con o o x sin espacio previo — genera conexiones circulares o cruzadas no deseadas
  • Los comentarios van con %% al inicio de la línea
  • No aplicar estilos con CSS externo — usar siempre classDef dentro del diagrama
  • mindmap y timeline no soportan classDef — usarlos solo cuando la estructura importa más que el color
  • Un diagrama por concepto — si se necesitan más, dividir en secciones separadas con texto entre ellas
  • El diagrama va siempre después del párrafo que lo introduce, nunca antes
  • xychart-beta aún es experimental — verificar que el renderizador lo soporte antes de usarlo
